Lab4: Traps

做完Lab4的收获：

  1. 我理解了trap（系统调用、异常、硬件中断）来自用户空间的trap的处理流程：uservec->usertrap->usertrapret->userret 。
    • 进入uservec之前，比如调用ecall指令时，硬件就已经完成了一些操作，切换到内核态、根据stvec存储的uservec进入uservec等。
    • uservec：主要是保存现场，并且切换到内核页表。
    • usertrap：根据trap的类型进行相应的处理。
    • usertrapret：trap处理完了，需要对trapframe等进行更新，外加其他处理操作。
    • userret：切换回用户页表，恢复现场。
  2. 知道了trampoline page中存放的是uservec和userret。
  3. trapframe在trap发生时进入内核态，进入trap向量uservec保存现场，切换页表，从userret切回页表，返回用户空间恢复现场。
  4. 用户空间进入内核空间需要保存现场，从内核空间调用用户空间的函数也需要另一个trapframe保存现场。

Trap主要有3种类型：系统调用、异常、硬件中断。Lab4主要是关注的是来自用户空间的trap。

来自用户空间的trap的工作流程，以系统调用为例：

用户程序进行系统调用，当前trap向量stvec是trampoline中的uservec，在ecall指令之后，从用户态切换到内核态，硬件会自动完成一些操作：设置epc为用户被中断指令的下一条指令的地址等，进入uservec，之后将寄存器值存储在trapframe中（保存现场），切换页表等，再进入usertrap()函数，根据sscuase寄存器的值判断trap的类型，如果是系统调用，就转到系统调用的处理，完成后usertrapret()，完成另外一些操作，后userret，切换页表，恢复用户程序的现场（pc [用epc恢复]，寄存器等），继续执行。

Task3 Alarm

Task3通过系统调用sigalarm(interval, handler)开启定时器，在到达时间后执行用户空间的handler函数，这是Task3的难点，因为时钟中断时，进程处于内核态，当前trapframe的epc并不是handler的地址，强行将trapframe的epc改成handler的地址返回用户态，确实可以执行handler，但是这一趟执行回来trapframe的内容全变了，因此，需要提前保存trapframe的内容到另一个trapframe中，在sigreturn中恢复原本的trapframe。在struct proc中增加alarm_trapframe，handler，interval以及定时需要的项。

一些提示：

  1. sys_sigalaram()应当将alaram interval和handler function pointer存到进程的结构新字段。
  2. 需要记录自从上一次调用进程alarm handler过了多少tick，你也需要在struct proc中增加该字段。你可以在allocproc()中初始化这些字段。
  3. 每个tick，硬件时钟会强制一个中断，这在kernel/trap.c中的usertrap()处理，每个tick，进程中记录tick的项加1，如果需要执行用户空间的handler函数，就先保存trapframe的内容到进程中新的一项trapframe alarm_trapframe，trapframe中的epc设为handler的地址，定时器的tick置零，然后返回用户空间执行，用户空间的handler最后执行sigreturn，再将trapframe恢复。
  4. 防止对handler的重复调用，如果一个handler没有return，内核不允许重新调用该handler，在进程中增加一项handler_running表示handler是否在运行。
