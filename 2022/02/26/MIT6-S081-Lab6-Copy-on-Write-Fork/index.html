Lab6: Copy-on-Write Fork

Lab6要求是先Copy-on-Write Fork，这是虚拟内存部分的又一个优化。该优化的出发点是xv6中的系统调用fork拷贝了父进程的所有用户空间内存到子进程，如果父进程很大，拷贝会使用很长时间，更糟的是这些工作通常是没用的，比如，fork之后接exec，子进程会丢弃之前拷贝的内存。另一方面，如果父进程和子进程使用同一个页面，一个或两者都写该页面，这时就需要拷贝。即copy-on-write fork的目标是延迟为子进程分配和拷贝物理内存页面，直到确实需要拷贝。

1. Copy-On-Write Fork

实验的提示：虚拟内存提供了一定程度的间接性，内核可以通过将PTE标记为非法的或者只读的阻止对内存的引用，导致page fault，且可以通过修改PTE改变地址的含义。计算机系统有一种说法，任何系统问题可以通过一定程度的间接性解决。懒惰页面分配提供了一个例子，这里的copy-on-write fork也是。我们可以使用PTE中的RSW位（第9,10位），这里我使用第9位，在riscv.h中定义宏#define PTE_S (1L << 8)，表示该页面是否是cow的页面。

![](https://cdn.jsdelivr.net/gh/wzmw-zr/blog-images/3-step virtual address translation.png)

copy-on-write fork为子进程仅仅创建页表，其中的PTE指向和父进程相同的物理页面，并且将父进程和子进程中的PTE设置为不可写，同时标记对应的RSW位置，这一步在拷贝用户页表的uvmcopy中实现。

COW fork使得用户内存的物理页面的释放更有技巧性，一个给定的物理页面可能被多个进程的页表引用，只有当最后的引用消失才释放。可以维护一个数组记录引用页面的进程数量。当kalloc时将页面引用设为1，进程fork会导致页面引用增加，进程从页表中删除PTE减少页面引用。这里我在kalloc.c中定义了一个page_ref数组记录页面的引用数，这个数组是临界资源（critical resource），所以对其访问、修改需要加锁，为了方便就直接使用kmem的lock。

2. allocate and copy page when write

在需要写被标记为COW的页面时，如果该页面的引用数为1，那么修改标志位即可，否则需要分配页面进行拷贝。会对用户内存进行写操作的行为：

  1. 用户空间修改操作作用在COW页面上，发生缺页中断，在usertrap中处理。（例子：*p = 1;）
  2. 内核空间修改操作作用在COW页面上，发生缺页中断，在kerneltrap中处理。
  3. sys_read，sys_pipe，copyout这种需要向用户空间写入内容的函数。
