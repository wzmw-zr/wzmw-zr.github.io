Lab 8: Locks

Lab 8需要重新设计内存分配器（Physical memory allocator）和缓存层（Buffer cache layer），提高并发程度。提高并发程度需要修改数据结构，并且使用合适的加锁策略减少冲突，避免死锁。

Lab 8带给我的收获是，提高并发能力，可以尝试细粒度锁，以及如何设计加锁策略避免死锁。

Task 1: Memory allocator

原本xv6的内存分配器在kalloc和kfree中使用全局的锁kmem.lock，为了提高并发能力需要使用细粒度锁，可以为每个CPU维护一个可用页面链表，以及一个锁，进程从当前的CPU获取页面，如果当前CPU没有空闲页面，就从其他CPU中找空闲的页面。

为了保证获取页面的过程是原子的，获取当前CPU对应的锁后，当前CPU没有空闲页面，从其他CPU找空闲页面，需要先加全局锁，然后对其他CPU分别加锁，解锁，这样是原子的，并且由于全局锁在中间，不会发生死锁的问题。

进程释放页面，就将页面加入进程运行在的CPU的可用页面链表中。

Task 2: Buffer cache

通常会有多个进程使用文件系统，但是buffer cache层只有一个全局锁bcache.lock，思路也是使用细粒度锁，但是buffer cache不像内存分配可以按CPU加锁，实验提示可以使用哈希表，对哈希表的bucket加锁，其余的细节和内存分配并发类似。
